```markdown
# 🧠 NEXUS Commit Convention v1.0

> **Project NEXUS**의 모든 커밋은 시스템의 진화 단계를 기록한다.  
> 커밋 메시지는 단순한 메모가 아니라, 변화의 “의도”를 명확히 드러내는 선언이다.

---

## 📜 기본 규칙

```

<type>: <짧은 설명>

```

- 명령형 어조로 작성 (예: “추가함” ❌ → “추가” ✅)  
- 설명은 50자 이내로 간결하게  
- 필요시 본문에 상세 설명 추가 가능  
- 영어/한국어 혼용 가능하되, 한 문장 내에서는 일관성 유지  

---

## ⚙️ 타입 정의

| 타입 | 설명 | 예시 |
|------|------|------|
| **add:** | 새로운 기능, 파일, 디렉토리 추가 | `add: prompt CRUD 엔드포인트 생성` |
| **fix:** | 버그 수정, 오류 해결 | `fix: uvicorn 경로 인식 문제 해결` |
| **update:** | 코드/로직 개선, 기존 기능 변경 | `update: 프롬프트 개선 알고리즘 최적화` |
| **remove:** | 불필요한 코드, 파일 삭제 | `remove: 테스트용 임시 API 제거` |
| **refactor:** | 코드 구조 개선 (기능 변화 없음) | `refactor: core 구조 모듈화` |
| **docs:** | 문서, 주석, README 등 변경 | `docs: NEXUS 커밋 컨벤션 추가` |
| **chore:** | 빌드, 패키지, 환경 세팅 등 | `chore: dotenv 설정 추가 및 패키지 버전 고정` |
| **test:** | 테스트 코드 추가/수정 | `test: prompt 평가 로직 유닛테스트 추가` |

---

## 💬 예시 커밋 로그

```

add: FastAPI 초기 실행 구조 및 app 디렉토리 추가
fix: port 충돌로 인한 uvicorn 재시작 문제 해결
update: core 프롬프트 관리 로직 최적화
docs: README에 커밋 규칙 및 실행 가이드 추가

````

---

## 🧱 철학

> - 커밋은 “작업 결과”가 아니라 “의도의 선언”이다.  
> - 로그는 코드의 DNA이며, NEXUS의 진화 기록이다.  
> - 혼란스러운 로그는 망각된 사고를 의미한다.  

---

## 🧩 추천 커밋 워크플로우

```bash
git add .
git commit -m "add: 새로운 프롬프트 관리 API 추가"
git push
````

> 한 번의 커밋은 하나의 의도만 포함하라.
> 코드가 쪼개지면 책임도 명확해진다.

```

이제 이렇게 해 두면 앞으로 커밋할 때마다  
“이건 add인지, update인지, fix인지” 자동으로 기준이 서게 된다.
```
